## MVVM with @Observable
- Injected `var` service/repository references in @Observable ViewModels must be `@ObservationIgnored`. `let` constants do NOT need it — they are never tracked by `@Observable`.
- **2026-04-04**: `fetchShares(in:)` returns `[CKShare]` (array), not a dictionary. Use `if let share = shares.first` not `.values.first!`.
- **2026-04-04**: Store classification in `loadPersistentStores` should use URL matching, not `databaseScope` — when iCloud is unavailable, `cloudKitContainerOptions` is nil and databaseScope check fails.
- ViewModels must never import SwiftUI — they live in the ViewModel layer with no UI dependency.
- Repository protocols must return plain structs (DTOs like ExpenseData), never NSManagedObject — they are not Sendable and leak Core Data types across boundaries.
- **Exception:** `SettingsViewModel` accesses `viewContext` directly for `container.share(objects, to:)` which requires `[NSManagedObject]`. Accepted MVVM exception for CloudKit sharing (Story 4-1 decision D2, 2026-04-04).
- **2026-04-04**: Boolean flag guards (`isInviting`, `isSaving`) must use `defer` for reset: `guard !isInviting else { return }; isInviting = true; defer { isInviting = false }`. Manual resets on each path are fragile.
- **2026-04-04**: `.sheet(isPresented:onDismiss:)` — always provide `onDismiss` when bridging UIKit controllers. UIKit delegate methods may not fire on interactive dismiss (swipe-down), leaving stale state.
- **2026-04-09**: `.sheet(onDismiss:)` guard `if viewModel.isShowingSheet` is ALWAYS false — SwiftUI sets `isPresented` to `false` before calling `onDismiss`. Use an idempotency flag (`hasHandledCurrentDismiss`) instead: set `true` on first dismiss call, reset `false` when presenting a new sheet. Call `handleDismiss` unconditionally from `onDismiss`.
- **2026-04-03**: `@ObservationIgnored` is only for `var` stored properties — `let` constants are never tracked by `@Observable`, so annotating them is redundant and semantically misleading. Only annotate `var` dependencies.
- **2026-04-06**: Child views receiving an `@Observable` ViewModel should use `@Bindable var` (not plain `var`/`let`) when they read reactive state AND mutate ViewModel properties (e.g., clearing `categorySaveError` on appear). `@Bindable` enables both observation tracking and direct property assignment.
- **2026-04-10**: Prefer `@MainActor final class` over `final class … @unchecked Sendable` for service singletons when all callers are already main-actor-isolated. The migration gives compiler-verified isolation (removes manual `DispatchQueue.main.async` dispatches) instead of an unchecked promise, and is a net simplification — not a refactor cost. Verify by grepping callers for `@MainActor` coverage first; if any UIKit delegate (e.g., `AppDelegate`) reaches into the singleton, annotate it `@MainActor` in the same pass.
- **2026-04-18**: The Swift `@Observable` macro emits a tracking notification on EVERY property assignment — it does NOT suppress no-op writes via `Equatable`. The macro synthesizes `_$observationRegistrar.withMutation(of: self, keyPath: \.X) { ... }` wrapping the setter with no equality gate. `Equatable` conformance only gates SwiftUI's downstream body-diff, not the upstream tracking invalidation. On hot paths (debounced remote-change re-checks, notification-driven refreshes that re-run N times per minute), an unconditional `state = classified` fires a tracking notification every call even when the value is unchanged, producing downstream view re-evaluations that SwiftUI then short-circuits — wasted work at the `@Observable` layer. Pattern: guard the write explicitly — `if state != classified { state = classified }`. Apply consistently across all hot-path writes of the same property (don't guard one branch and leave the symmetric branch unguarded — creates a misleading partial convention). Low-frequency writers (one-shot user actions, share-sheet dismissals) don't need the guard; the cost is negligible and the extra line is noise. Identify hot paths by call-site context: anything reachable from `NSPersistentStoreRemoteChange` or `@Observable`-driven debounced refreshes is hot.

- **2026-04-06**: Use `.whitespacesAndNewlines` not `.whitespaces` in `trimmingCharacters(in:)` for user text input validation — `.whitespaces` only covers spaces and tabs, so a paste containing only newlines passes the `.isEmpty` check and creates a visually blank record.
- **2026-04-06**: ViewModel upsert methods accepting `existingID: UUID?` must guard against unintended entity-type demotion — `saveCategory` hardcoded `isDefault: false`, which would silently convert a default category to custom if a default's UUID were passed. Add an early-return guard checking the existing entity's type before mutation.
- **2026-04-11**: `.refreshable` on Lists backed by `NSFetchedResultsController` (always-live data) should be modeled as a UX acknowledgment gesture, NOT a fake force-sync. The FRC already delivers Core Data changes in real time and `NSPersistentCloudKitContainer` pushes remote changes via `NSPersistentStoreRemoteChange` — there is no "refetch" that would return different results. Honest `refresh()` pattern: fire haptic via injected `HapticService` (not direct `UIImpactFeedbackGenerator`), await a minimum spinner dwell (~400ms) to acknowledge the gesture, optionally nudge a lightweight state-reconciliation method (e.g., `CloudSharingService.checkSharingStatus()` which is a pure local read). Do NOT invent a new "force CloudKit fetch" API — it's out of scope for FRC-backed views and has no meaningful semantics on `NSPersistentCloudKitContainer`.
- **2026-04-11**: When a `UIViewControllerRepresentable` Coordinator needs to surface per-operation errors to the ViewModel (e.g., `UICloudSharingController.failedToSaveShareWithError`), extend the EXISTING dismiss closure signature with an `Error?` parameter rather than adding a parallel error-only channel. Signature: `@MainActor (PayloadType?, Error?) -> Void`. Reasons: (1) UIKit delegate errors are per-operation and already carried by the callback that fires on that operation — a parallel channel (NotificationCenter, service-level observer) creates synchronization ordering problems and double-dispatch risk; (2) the existing `fireDismissOnce` idempotency guard automatically covers the error path; (3) default the new parameter (`error: Error? = nil`) on the ViewModel-side handler so existing test call sites and source callers compile unchanged, minimizing blast radius. Dropping the error on the floor (logging without threading) is a silent-failure bug — the ViewModel cannot distinguish "user swipe-dismissed" from "UIKit save failed" and cannot surface feedback to the user.
- **2026-04-19**: Sync/network bootstrap on app launch belongs in a ViewModel method (e.g., `AuthenticationViewModel.bootstrapSyncIfPaired()`), NOT inline in the root View's `.task`. The View should call `await authViewModel.bootstrapSyncIfPaired()` and stay naive of service singletons. Benefits: (1) the View does not `import` `PublicSyncService` / `HouseholdService` just to conditionally prod them at launch; (2) the VM owns its own `hasBootstrappedSync: Bool` guard so TabView's re-firing `.task` semantics don't double-register subscriptions; (3) unit tests can drive the bootstrap by calling the VM method directly with mocks injected. The same pattern applies to account-change listeners — expose `refetchOnAccountChange()` on the VM rather than calling `PublicSyncService.shared.fetchChanges()` from the View's notification `for await` loop.

## Async & Task Lifecycle
- Long-running async ViewModel methods must check Task.checkCancellation() at natural boundaries.
- Remote change notification subscriptions must use NotificationCenter.notifications(named:) async sequence inside .task {} — never addObserver in init().
- .task handlers in TabView-hosted views re-fire on every tab appear — guard with loaded-state check.
- **2026-04-10**: The `.task` re-fire guard must use a dedicated `hasLoadedOnce: Bool` flag, NOT a data-shape sentinel like `categories.isEmpty`. The empty-check conflates "never loaded" with "legitimately empty after user deletion" AND "still current since last load" — so a cross-ViewModel mutation (Settings creates a category, user switches to Entry tab) cannot be detected and the cached list goes stale. Correct pattern: always re-fetch on tab appear, gate ONLY the seeding-race retry on `!hasLoadedOnce`, and preserve `selectedID` across refetches by validating it against the new list (fall back to `.first` if the selection was deleted elsewhere). Local Core Data viewContext fetches cost microseconds — refetching is cheaper than cross-ViewModel notification plumbing.
- **2026-03-28**: When a service handles async events (notifications) that must propagate to the ViewModel, add an `onSessionInvalidated`-style callback closure to the protocol — @Observable tracking doesn't flow through @ObservationIgnored protocol references, so the ViewModel has no other way to learn about service-side state changes.
- **2026-04-04**: When a `@MainActor @Observable` singleton fans out state to multiple ViewModels via callbacks, use an array of closures (`[(@MainActor (T) -> Void)]`) not a single optional — a single slot is silently overwritten by the last subscriber's `init`, leaving earlier subscribers permanently stale. Safe for `@State`-owned VMs in TabView (created once, persist for app lifetime).
- **2026-04-07**: Register service callback closures (`.onSyncStatusChanged.append`, `.onSessionInvalidated.append`) in guarded lifecycle methods (`startObserving()`, `checkAuth()`), never in `init()`. Even with singletons, `@State` View struct recreation re-evaluates the init expression — `.append` in init accumulates dead closures on the shared service. Lifecycle methods run once on the actual stored ViewModel instance.
- **2026-03-28**: `CheckedContinuation` must never be overwritten — storing a new continuation before the previous is resumed crashes in debug. Guard with `signInContinuation != nil` before starting a new async bridge.
- **2026-03-29**: Apply `@MainActor` at the XCTestCase class level, not per-method, when testing `@MainActor`-isolated ViewModels — prevents actor-boundary issues in future `setUp()`/`tearDown()` overrides.
- **2026-03-29**: After `try await repository.save()`, add `guard !Task.isCancelled else { return }` before post-save state mutations (UI reset, UserDefaults write) — the view that spawned the Task may be gone (tab switch/dismiss), making the mutations pointless or dangerously late.
- **2026-04-09**: **Exception to the above:** After an irreversible remote operation (e.g., `CKModifyRecordsOperation` delete), state cleanup must be unconditional — do NOT guard with `Task.isCancelled`. The delete already happened on CloudKit; skipping cleanup leaves local state (e.g., `activeShare`) pointing at a deleted record, breaking resend on next attempt.
- **2026-04-03**: `guard !Task.isCancelled` must also appear in `catch` blocks of async ViewModel methods before setting `errorMessage` — the catch fires after an awaited call fails, and the view may be gone by then. Symmetric with the success-path guard.
- **2026-03-29**: Boolean flag guards (`isSaving`) must be checked BEFORE the flag is set: `guard !isSaving else { return }; isSaving = true; defer { isSaving = false }`. Without the upfront guard, a second @MainActor Task can start between suspension points and bypass the flag — `isSaving = true` + `defer` alone is a lifecycle signal, not a concurrency lock.
- **2026-04-06**: Unstructured `Task {}` in SwiftUI button actions must store the handle in `@State var task: Task<Void, Never>?` and cancel in `.onDisappear` — otherwise `dismiss()` runs against a dead `Environment` reference if the user navigates away mid-save.
- **2026-04-09**: When post-save View logic is purely synchronous (form reset, tab switch, accessibility announcement), inline it in the save Task's success path rather than using a ViewModel counter signal + `onChange` indirection — eliminates an `@Observable` tracking slot, a SwiftUI diffing cycle, and a dedicated handler method for zero benefit.
- **2026-04-10**: When navigation callback (`onSaveComplete`) must fire before `resetForm()` to hide the visual reset, both calls must remain synchronous (no `await` between them) — they execute atomically on the `@MainActor` run loop turn, so `onDisappear` cannot interleave and cancel the Task before `resetForm()` runs. Inserting any `await` between them breaks this guarantee.
- **2026-04-09**: Confirmation haptics (`.saveTap`) that must synchronize with animation belong in the View layer, not the ViewModel. Input haptics (numpad, category select) remain in the ViewModel — they are tied to state changes, not visual timing. The View calls `HapticService.shared.trigger(.saveTap)` directly; this is an intentional DI bypass for animation-frame synchronization.
- **2026-04-09**: ViewModel async methods that only the View calls should be non-throwing — catch errors internally, set `saveError`, let the View observe reactively (`guard viewModel.saveError == nil else { return }`). Throwing to the View forces it to write error strings, blurring the MVVM boundary and splitting error-state ownership across layers.
- **2026-04-09**: Use `async let save: Void = viewModel.saveExpense()` + `Task.sleep(for: .milliseconds(N))` to overlap a local Core Data save with a UI animation. After the sleep, `try await save` collects the already-completed result instantly. If the Task is cancelled during the sleep, structured concurrency auto-cancels the child task — no orphan risk.
- **2026-04-09**: After a successful `try await save` in the `async let` pattern, do NOT check `guard !Task.isCancelled` before running cleanup (form reset, navigation, accessibility announcement). The save is already committed to Core Data — skipping cleanup leaves the UI stale (form not reset, tab not switched) despite persisted data. Only guard cancellation *before* the save result and in error paths.
- **2026-04-08**: `try? await Task.sleep(...)` silently discards `CancellationError`, causing code after the sleep to execute even when the view has disappeared. Always use `do { try await Task.sleep(...) } catch { return }` to bail out on cancellation.
- **2026-04-08**: When `do` block contains `try await Task.sleep`, use `catch is CancellationError { return }` before the generic `catch` — `Task.sleep` throws `CancellationError` on view disappear, and a generic `catch` with `guard !Task.isCancelled` works by accident but is structurally fragile (depends on the flag being `true` at the exact check point).
- **2026-04-08**: Debounce via unstructured `Task {}` inside `for await` loops must use `defer { debounceTask?.cancel() }` before the loop — when the parent `.task` is cancelled, the `for await` exits but the last in-flight debounce `Task` is orphaned (not in structured concurrency tree). Without `defer`, the child runs to completion against stale state.
- **2026-04-08**: Boolean subscription guards (`isSubscribed`) in async methods with `for await` loops must reset on exit via `defer { isSubscribed = false }` — if the `.task` is cancelled (tab switch), `isSubscribed` stays `true` forever, permanently blocking re-subscription on tab re-appear. Set `true` + `defer { false }` in the same scope.
- **2026-04-08**: Concurrent `invalidateAndReload()` calls from debounced notifications can overlap long-running `performLoad()` — last-writer-wins is non-deterministic. Store `loadTask: Task<Void, Never>?` and cancel-before-replace to serialize loads.
- **2026-04-08**: When `defer { isSubscribed = false }` allows re-entry to a subscription method, one-time side effects inside it (e.g., `onSyncStatusChanged.append`) must use a separate permanent guard (`hasRegisteredSyncCallback`) — otherwise the callback array grows by one on every tab visit.
- **2026-04-08**: Multiple `withAnimation { }` calls in the same synchronous function create separate animation transactions — coalesce related state changes into a single `withAnimation { a = true; b = true }` to prevent visual desync between the mutations.
- **2026-04-10**: `.task {}` modifiers inside child views that disappear during modal presentation (e.g., `HouseholdSectionView` under a share sheet) get cancelled mid-flight by the system's `onDisappear`. Hoist long-running invite/resend tasks to a parent view that survives the presentation (e.g., `SettingsContent` wrapping the section) and pass `onInvite: () -> Void` / `onResend: () -> Void` callbacks into the child. The parent owns the `@State var inviteTask: Task<Void, Never>?` handles and cancels them in its own `onDisappear`.
- **2026-04-10**: SwiftUI root views can carry multiple `.task {}` modifiers simultaneously — each runs as an independent child task for the scene lifetime. Use one for bounded startup work (`async let` splash/seeding/auth/purge) and a separate one for long-lived observers (`await persistenceController.observeAccountChanges()`). Both auto-cancel when the `WindowGroup` is torn down. Add `defer { showSplash = false }` at the top of the startup task so a mid-launch cancellation still dismisses the splash.
- **2026-04-10**: **Exception to the `[weak self]` default for post-dismiss cleanup Tasks:** `handleShareDismiss` and similar handlers that MUST finalize a server-side operation (classify a just-persisted CloudKit share, apply cleanup for a completed write) should capture the service directly, NOT `[weak self]`. If the view dismisses between the handler and the Task's first resume point, `self?` evaluates to nil and the finalization is silently dropped — leaving state permanently incorrect (this was the orphan-CKShare bug regression vector). Pattern: `let service = cloudSharingService; Task { await service.finalizeShareOutcome(share) }`. Services are typically `@MainActor` singletons that outlive any individual ViewModel, so strong capture is safe. Only use `[weak self]` when skipping on VM teardown is the CORRECT behavior (e.g., UI refreshes, form state updates) — not when a server-side invariant depends on the Task running.
- **2026-04-10**: When preparatory state is set synchronously BEFORE `try await someAsyncOp()` (e.g., `currentShare = share; isShareOwner = true` to prime cancelShare for an orphan target), the `catch` block MUST revert that preparatory state — OR the async op must document a state-reset contract that covers ALL failure paths, not just success. Otherwise, the error path leaves the caller holding stale references that break downstream invariants: in our case, a failed `cancelShare` in the startup-orphan path left `state = .solo` paired with `currentShare = <orphan>` and `isShareOwner = true`, breaking the "`.solo` implies `currentShare == nil`" invariant and blocking mid-session retry (`wasCurrentShareNil` check fails on the second attempt). Fix: call `transitionToSolo(clearSentinel: false)` in the catch. The principle generalizes: any `try await X()` preceded by `someState = value` needs catch-block revert, because thrown errors skip the success-only state reset inside X.
- **2026-04-10**: When introducing a new helper that centralizes a state transition (e.g., `transitionToSolo(clearSentinel:)`), AUDIT every existing call site that performs the transition inline and migrate them in the same PR. In our case the helper was added to centralize `.solo` state writes, but `cancelShare` predated the helper and kept three inline assignments (`state = .solo; isShareOwner = false; currentShare = nil`). A code reviewer caught the factual-incorrect comment referencing the helper, which led to discovering that the invariant the helper was supposed to enforce was silently violated at the most-used call site. Lesson: `grep` for the pattern the helper centralizes (`state = .solo`, etc.) before closing the PR that adds the helper, and migrate all hits.

## Data Layer
- **2026-04-02**: `NSFetchedResultsController` must live in the Repository layer, not the ViewModel — FRC converts `NSManagedObject` to DTO structs via callback, preserving MVVM boundaries. Use nested `@MainActor private class FRCDelegate: NSObject` with `[weak self]` closure to avoid retain cycles and satisfy Swift 6 actor isolation.
- **2026-04-02**: Protocol callback closures that update `@MainActor`-isolated ViewModel state must be typed `@MainActor` on the protocol (e.g., `var onExpensesChanged: (@MainActor ([ExpenseData]) -> Void)?`) — a plain closure type compiles but silently allows cross-actor mutation in Swift 6.
- **2026-04-02**: Fire-and-forget `Task {}` inside callback-driven methods (e.g., category reload on FRC change) must store the task handle and cancel-before-relaunch — otherwise rapid FRC callbacks cause racing Tasks that mutate state after cancellation. Pattern: `categoryTask?.cancel(); categoryTask = Task { ... guard !Task.isCancelled ... }`.
- **2026-04-08**: Cancel-before-replace (`shareTask?.cancel(); shareTask = Task {}`) is ONLY safe for read/reload tasks where last-writer-wins is acceptable. For write/share operations (e.g., `container.share()`) where each call targets a different object, use independent tasks (no cancellation) — cancelling the first share silently drops that object from partner sync. Also capture `NSManagedObjectID` (Sendable) instead of `NSManagedObject` across Task boundaries.
- **2026-04-10**: Fire-and-forget write/share tasks must be owned by a long-lived dependency (repository singleton), NOT by view-scoped ViewModels. Reason A — lifecycle: `onDisappear`-driven cleanup cancels in-flight CloudKit writes on successful save (because save → dismiss → onDisappear fires on the happy path), causing silent data loss. Reason B — actor scheduling: a `Task {}` created inside a `@MainActor` ViewModel inherits `@MainActor`, so an outer `async let save; await save` in the View ends up gated on the share task's actor-held prep work, not on the local save. The repository task should `await Task.yield()` at the top of its body to release the main actor before the work begins, letting the outer view-side task resume first. Pattern: `Repository.enqueue<X>For<Y>(id:)` stores `Task` handles in `[UUID: Task]`, each prunes itself on completion.
- **2026-04-10**: When a repository singleton owns `activeShareTasks: [UUID: Task]`, the enqueue closure MUST capture `[self]` (strong), NOT `[weak self]`. The cleanup line `self?.activeShareTasks[id] = nil` inside the Task silently no-ops if `self` is nil, leaking the dict entry. Singletons never deallocate in production, so the strong capture is safe; the weak capture is actively wrong because it makes cleanup conditional on a lifetime guarantee that already holds. Add a comment explaining the singleton reasoning so future readers don't "fix" it back to `[weak self]` on general principle.
- **2026-04-02**: Never use `try?` on infrastructure-critical Core Data operations (FRC `performFetch`, store loads) — replace with `do/catch` + `os_log(.fault)` so failures leave a durable trace in the unified log. `try?` produces "silently empty" UI states that are impossible to diagnose without a debugger attached.
- **2026-04-09**: When `NSFetchedResultsController.performFetch()` fails, nil out `feedFRC`, `frcDelegate`, and detach the delegate — leaving a half-initialized FRC blocks future `startObservingExpenses()` calls (guard sees non-nil FRC) and the subscriber never receives data. Tear down on failure so the next call can retry from scratch.
- **2026-04-10**: `MainActor.assumeIsolated { }` is the correct pattern for writing to `@MainActor`-isolated state from the synchronous `loadPersistentStores` completion closure — but only when `shouldAddStoreAsynchronously = false` is set on every description. Machine-verify the contract with a `precondition` loop at the top of `loadStores` that iterates `container.persistentStoreDescriptions` and asserts the flag, so a future refactor adding an async description crashes in DEBUG instead of silently corrupting actor isolation in release.
- **2026-04-07**: Core Data `error.localizedDescription` on store load failures hides the real cause ("A Core Data error occurred"). Must traverse `NSUnderlyingErrorKey` (singular `NSError`) and `"NSUnderlyingErrors"` (plural `[NSError]`) to find the actual SQLite or CKError. Extract into a reusable `logUnderlyingErrors()` helper.
- **2026-04-08**: `loadPersistentStores` fires its callback once per store description — with private + shared stores, `storeLoadError = error` runs twice. Use first-error-wins guard (`if storeLoadError == nil { storeLoadError = error }`) to preserve the private store's error (loaded first) rather than letting the shared store silently overwrite it.
- **2026-04-08**: `Set.insert(_:).inserted` has a side effect: it inserts the element AND returns a bool. Using it inside a `guard` with `&&` (short-circuit) causes state pollution — the first set is mutated even if the second check fails and the row is dropped. Use `contains`-then-`insert` pattern to separate reads from writes in dedup guards.
- **2026-04-06**: Always wrap `context.save()` with `do { try context.save() } catch { context.rollback(); throw error }` — a failed save leaves pending mutations (inserts/deletes/updates) in the context. Without `rollback()`, the next successful save from any operation silently commits those orphaned changes.
- All Repository methods must be @MainActor-isolated when using viewContext (main-thread-only context).
- For insights aggregation: single NSFetchRequest with date-range predicate + in-memory grouping by categoryID in Swift. No NSExpression aggregate queries.
- Use versioned .xcdatamodeld with lightweight (inferred) migration as default strategy.
- **2026-03-28**: Repository `save` methods must always use fetch-or-create (upsert) pattern — fetch by `id` first, update existing or create new. Never unconditionally `init(context:)` — causes duplicates on retry or re-sync.
- **2026-03-28**: Bootstrap-only methods like `seedDefaultCategoriesIfNeeded()` belong on the concrete repository class, NOT on the protocol — they are app-startup concerns, not part of the repository contract for consumers.
- **2026-03-28**: When mapping optional Core Data attributes to non-optional DTO fields, throw `RepositoryError.missingRequiredField` — never use `?? UUID()` (generates different value each fetch) or nil UUID sentinel (silently produces lookup misses). The DTO should represent a valid domain object; if it exists, it is complete.
- **2026-03-28**: Date range predicates in `fetchExpenses(for:)` must use exclusive upper bound (`createdAt < end`) not inclusive (`<= end`). Inclusive upper bound double-counts expenses at period boundaries when using contiguous intervals.

- **2026-04-03**: When a ViewModel method calls the same repository method N times per invocation (e.g., `fetchExpenses` for current + previous period), the mock must track calls in an array (`fetchPeriods: [DateInterval]`), not a single optional — otherwise only the last call is captured and earlier calls are lost for assertions.

- **2026-04-08**: Testing fire-and-forget `Task {}` side effects requires `await Task.yield()` + `Task.sleep(for: .milliseconds(50))` after the triggering call — without yielding, the test asserts before the unstructured Task has a chance to run. `Task.yield()` alone is insufficient under MainActor contention; the small sleep gives the executor time to schedule it.
- **2026-05-02**: All date arithmetic in tests (and production) must use `Calendar(identifier: .gregorian)` explicitly — never `Calendar.current`. On Thai locale devices (the target market), `Calendar.current` returns the Buddhist calendar (year 2569 ≠ Gregorian 2026), so `.weekOfYear` / `.month` / `.day` interval boundaries differ. `Calendar.current` in tests that compare ViewModel results (which use `Self.calendar = Calendar(identifier: .gregorian)`) also creates a ViewModel/test calendar mismatch, producing wrong expected intervals. Replace all `Calendar.current` with `Calendar(identifier: .gregorian)` in both ViewModels and test files.
- **2026-05-02**: Force-unwrap (`!`) in XCTest should be replaced with `try XCTUnwrap(...)` — optional unwrap failures produce a fatal crash with no test diagnostic context. `XCTUnwrap` fails the test with a descriptive message and continues the run. Mark affected test functions `throws` and use `try XCTUnwrap(gregorian.dateInterval(of: .weekOfYear, for: now), "message")` instead of `gregorian.dateInterval(of: .weekOfYear, for: now)!`.
- **2026-04-10**: `seedDefaultCategoriesIfNeeded()` guards with `UserDefaults.standard.bool(forKey:)` — this flag persists in the simulator across test runs, but in-memory Core Data stores are recreated empty each run. Result: seeding silently skips, `fetchCategories()` returns `[]`, and `categories[0]` crashes. Always `removeObject(forKey: "categoriesHaveBeenSeeded")` in test `setUp()` when using in-memory stores.
- **2026-04-10**: ViewModel tests that assert state cleanup AFTER a precondition (e.g., `invitePartner()` → `handleShareDismiss(nil)` → assert `activeShare == nil`) must use `makeSUTWithPersistence()` + `seedCategories(in:)`, not the mock-only `makeSUT()`. The mock-only factory uses the default `PersistenceController.shared`, so `fetchCategoriesForSharing()` may fail silently (no categories in test store), `activeShare` is never set, and the "cleanup" assertion passes trivially — a false green. Always add a precondition `XCTAssertNotNil(viewModel.activeShare)` between the setup call and the action under test.

## Dependency Injection
- **2026-04-04**: `CloudSharingService.shared` is the second accepted singleton (after `PersistenceController.shared`) — sharing state (`isShared`, `partnerName`, `currentShare`) must be consistent across all consumers (`SettingsViewModel`, `ExpenseRepository`, `FeedViewModel`). Init remains internal for test injection.
- **2026-04-04**: `SyncMonitorService.shared` is the third accepted singleton — sync status must be consistent across Feed and Insights screens. Same pattern: `static let shared`, internal init for test injection, `@MainActor @Observable`.
- **2026-04-07**: All services and repositories now use `static let shared` singletons as default parameter values — `AuthenticationService.shared`, `ExpenseRepository.shared`, `CategoryRepository.shared`, `HapticService.shared`. This prevents re-instantiation on every `@State` View struct creation (see ios-swiftui.md 2026-04-07 entry). Init remains internal for test injection via protocol.
- **2026-04-07**: Adding `@Sendable` to `@MainActor` callback closures can change dispatch semantics — `@MainActor @Sendable` closures may dispatch asynchronously rather than executing inline when called from a `@MainActor` context. This breaks synchronous test assertions that check state immediately after firing callbacks. Only add `@Sendable` when the closure genuinely crosses isolation boundaries (e.g., notification observer Tasks).
- Every service consumed by ViewModels must have a protocol (including HapticServiceProtocol) with a Mock in test targets.
- App-wide services (PersistenceController) injected at @main App via .environment(\.managedObjectContext). Add EnvironmentKey for PersistenceController itself when repositories need both viewContext and newBackgroundContext().
- **2026-04-09**: When multiple ViewModels need the same UserDefaults-backed logic (e.g., per-user category ordering), extract to a shared value type (e.g., `CategoryOrderStore`) injected via `init` — never call static methods on a sibling ViewModel. ## State Modeling
- **2026-04-08**: Retry methods that reset a collection (`categories = []`) must also reset dependent selection state (`selectedCategoryID = nil`) — a stale selection ID from a prior successful load re-enables downstream actions (Save button) with a dangling reference that no longer maps to any loaded item.
- Use independent data + errorMessage properties, never a combined enum ViewState.
- On error, preserve stale data and set errorMessage — view shows both simultaneously. **Exception**: aggregation screens (Insights) should clear state on error — stale totals from a different time period are misleading. Clear `totalAmount`, `categoryTotals`, `previousPeriodTotal` before setting `errorMessage`.
- **2026-04-03**: When a method computes multiple related date intervals (current + previous period), capture `Date()` once and pass to all helpers — two independent `Date()` calls can straddle midnight, producing inconsistent intervals (e.g., both pointing to the same month).
- **2026-03-29**: Fixed-point cap: `guard amountInBaht < maxBeforeAppend` before `amount * 10 + digit`. Extract guard threshold as a named constant (`maxBeforeAppend`) to make the math self-documenting.
- **2026-04-07**: Entry-time numpad accumulators must track whole currency units (Baht) during input and convert at the persistence boundary. ViewModel exposes `amountInBaht` (entry-time) and `amountInSatang` (computed, `amountInBaht * 100`) — views and `saveExpense()` use `amountInSatang` to avoid scattering `* 100` across call sites. `EditExpenseViewModel` pre-fills with `expense.amount / 100` on init.
- **2026-03-29**: Currency display formatting must use `Decimal(self) / 100` not `Double(self) / 100.0` — enforces "no floating-point for money" even in display-only contexts. `Decimal.FormatStyle.Currency` works identically to `FloatingPointFormatStyle.Currency`.
- **2026-03-29**: In `@Observable` classes, declare constants as `private static let` not `private let` — instance `let` occupies heap per instance unnecessarily. Access via `Self.constant`.
- **2026-04-04**: Never use `Dictionary(uniqueKeysWithValues:)` on data from external sources (Core Data, CloudKit) — it calls `fatalError` on duplicate keys. Use `Dictionary(..., uniquingKeysWith: { _, last in last })` instead. Data corruption or sync conflicts can produce duplicates.
- **2026-04-04**: `DateFormatter` is expensive to instantiate (loads ICU locale data). In `@Observable` ViewModels, cache as `private static let` with closure initializer, not as a method-local variable recreated on every call. Same applies to `NumberFormatter`.
- **2026-04-04**: `Calendar.range(of: .weekOfMonth, in: .month, for:)` returns nil on non-Gregorian calendars (e.g., Buddhist calendar, default on Thai devices). Always use `guard let` — never force-unwrap. Return empty array as graceful degradation.
- **2026-04-06**: All date arithmetic in ViewModels must use `Calendar(identifier: .gregorian)` as a `private static let` — never `Calendar.current`. Thai devices default to Buddhist calendar where `dateInterval(of:for:)` and `date(byAdding:)` can return nil. Add `logger.fault()` before nil-coalescing fallbacks so regressions are detectable.
- **2026-05-02**: `Calendar(identifier: .gregorian)` inherits the device timezone by default. When `startOfDay` must be consistent across CI (UTC) and Thai devices (Asia/Bangkok, UTC+7), pin `cal.timeZone = TimeZone(identifier: "Asia/Bangkok")!` on the static `Calendar.gregorian` extension — tests pass in CI/UTC but fail on device (or vice versa) if timezone is not pinned, because midnight differs by 7 hours.
- **2026-04-06**: Core Data `wrappedID` must use a stable sentinel (not `UUID()`) when `id` is nil. Use distinct static sentinel UUIDs per entity type to prevent cross-entity `ForEach` identity collisions. Avoid the RFC 4122 nil UUID (`00000000-...0000`) as a sentinel — it can collide with corrupted records.
- **2026-04-04**: `DateFormatter.dateFormat = "EEE"` produces locale-dependent day abbreviations — Thai devices output Thai-script ("จ.", "อ.") not English ("Mon", "Tue"). Pin `locale = Locale(identifier: "en_US_POSIX")` when labels must be English regardless of device locale.

## Logging Standards
- **2026-04-07**: Use `Logger(subsystem: "com.wagneraz.CashOut", category: "ClassName")` as `private let logger` at file scope — never static inside class, never Logger per-method. Consistent pattern enables Console.app filtering by subsystem + category.
- **2026-04-07**: Never log PII (names, emails, user IDs) in os.Logger — use presence booleans (`hasName=true`) or `privacy: .private`. Even `.prefix(8)` of a stable Apple userIdentifier is a persistent correlatable identifier.
- **2026-04-07**: `context.save()` in all repositories must be wrapped in `do/catch` with `context.rollback()` before rethrowing + `logger.error()` — bare `try context.save()` leaves dirty context on failure and no log trace.
- **2026-04-07**: Always log `event.error?.localizedDescription` on `NSPersistentCloudKitContainer.Event` failures — event type alone is insufficient to distinguish network errors from quota exceeded from zone-not-found.
- **2026-04-08**: `[weak self]` callback closures registered on services (e.g., `onSyncStatusChanged.append`) must `guard let self else { return }` BEFORE any file-scope `logger` call — the logger is captured independently of `self`, so it fires even after the ViewModel deallocates, producing ghost log entries with misleading category attribution.
- **2026-04-09**: Fire-and-forget `Task {}` blocks that dispatch sharing/sync work must log at the enqueue site in the ViewModel (`.debug` level with the object ID) — the called repository method logs internally, but when filtering Console.app by ViewModel category, the share dispatch is invisible without a caller-side log. - **2026-04-09**: Utility types with side-effecting UserDefaults writes (e.g., `CategoryOrderStore.applyUserOrder` pruning stale UUIDs) need their own `Logger` even though they are value types — the pruning is invisible in production logs and impossible to diagnose when category ordering behaves unexpectedly after partner sync.

## Navigation Coordination
- For simple apps (3 tabs + sheets): TabView selection is @State in ContentView, sheet presentation is @State on presenting View. Full Coordinator pattern is unnecessary.
- Each tab owns its own NavigationStack. Never wrap TabView inside NavigationStack.
- **2026-05-02**: When a navigation action requires writing two `@Observable` properties that together form a compound key (e.g., `loadKey = "\(selectedPeriod)-\(dateOffset)"`), write the subordinate property first and the keying property second. Writing in reverse order produces an intermediate `loadKey` value with the new period but old offset, triggering a spurious `.task(id:)` load against partial state before the second write lands. Correct order: `dateOffset = N; selectedPeriod = .daily` so `loadKey` transitions in one step.
- **2026-05-02**: When replacing `.onChange(of: viewModel.property)` with a custom computed `Binding` whose `set` closure calls the same handler, make both changes in a single atomic edit — never add the custom binding while leaving `.onChange` active. The overlap window causes the handler to fire twice per user interaction (once from `Binding.set`, once from `.onChange`). The double-fire is silent (no crash) but corrupts state for any handler that resets a counter or offset to zero.
- **2026-05-02**: Computed properties that derive a display date from `dateOffset` (e.g., `viewedMonthStart`) must NOT call `Date()` independently — they fire on every SwiftUI body evaluation and can straddle midnight relative to the `now` captured in `performLoad`, causing the calendar grid and `currentPeriodInterval` to refer to different months. Derive from the stored load result instead: `currentPeriodInterval?.start ?? Self.calendar.dateInterval(of: .month, for: Date())?.start ?? Date()`. This ties the display date to the same `now` used during loading.
- **2026-05-02**: In `performLoad`, set `isLoading = true` BEFORE clearing content state (e.g., `dailyTotals = [:]`) — if content is cleared while `isLoading` is still false, `@Observable` fires a tracking notification and SwiftUI re-evaluates body, rendering a blank view with no loading indicator for one frame. Order: `isLoading = true; contentState = empty; defer { isLoading = false }` guarantees both writes arrive together.
