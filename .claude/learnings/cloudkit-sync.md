# CloudKit Sync & Sharing

<!-- One line per learning, brief and actionable -->

## Shared Database Setup
- SwiftData does NOT support CloudKit shared databases (confirmed through iOS 26). Must use Core Data + NSPersistentCloudKitContainer.
- NSPersistentCloudKitContainer requires two NSPersistentStoreDescription configs: private scope + shared scope.
- The shared zone is framework-managed via `NSPersistentCloudKitContainer.share(_:to: nil)` — do NOT create a raw `CKRecordZone("HouseholdZone")`. The framework names the zone internally.
- Info.plist must contain `CKSharingSupported = true` or partner join callbacks silently fail.
- Zone existence must be re-verified on every fresh launch — users can delete zones via iOS Settings → iCloud.
- **2026-04-06**: Never `fatalError` in `loadPersistentStores` callback — use `logger.fault` + `storeLoadError` property so the app degrades to a non-functional but non-crashing state. Guard downstream operations (e.g., `purgeOldHistory`) on `storeLoadError == nil`.
- **2026-04-08**: Crash-prevention guards for Core Data store availability (e.g., preventing `NSInternalInconsistencyException` on `context.save()` with zero stores) belong inside the repository method itself, not at the call site. Inside the repository, check `privatePersistentStore != nil` when the operation targets private data (e.g., seeding default categories) — this is more precise than `coordinator.persistentStores.isEmpty` which passes when only the shared store loaded. Caller-side guards in the app startup code are fragile and should be avoided.
- **2026-04-06**: `CKAccountChanged` may fire on an arbitrary thread — always dispatch store reference mutations to `DispatchQueue.main.async` to avoid TOCTOU races with `@MainActor`-isolated callers.
- **2026-04-07**: Never call `loadPersistentStores` twice on the same `NSPersistentCloudKitContainer` — the coordinator retains partially-initialized state from the failed attempt. Use a static factory (`configuredContainer()`) and create a fresh container instance for the retry path.
- **2026-04-07**: When destroying Core Data store files for DEBUG recovery, delete `-ckAssets` directory alongside `-wal`/`-shm` — stale CloudKit metadata from a destroyed store triggers `changeTokenExpired` on next sync.
- **2026-04-06**: When re-validating a cached `CKShare` via `fetchShares(in:)`, keep the cached share on transient fetch errors (network, etc.) — discarding it creates a duplicate share zone via `container.share(objects, to: nil)`. Only clear when the fetch succeeds and the share is confirmed missing.
- **2026-04-10**: Shared store `databaseScope = .shared` must be set on a local `NSPersistentCloudKitContainerOptions` constant BEFORE assigning to `sharedDesc.cloudKitContainerOptions`. The fragile `desc.cloudKitContainerOptions?.databaseScope = .shared` optional-chain mutation works today only because the options class is a reference type — a future SDK change to the property getter could silently route the shared store to the private CloudKit database scope with no crash and no error log.
- **2026-04-10**: Set `shouldAddStoreAsynchronously = false` explicitly on every store description and add a `precondition` loop at the top of the `loadStores` helper that asserts the flag — `NSPersistentCloudKitContainer`'s synchronous-callback behavior is not formally documented, so `MainActor.assumeIsolated { }` inside the completion handler depends on an implicit contract. The precondition machine-verifies it so a future refactor that drops the flag crashes loudly in DEBUG instead of silently corrupting actor isolation.
- **2026-04-10**: `observeAccountChanges` must reset `storeLoadError = nil` alongside `privatePersistentStore = nil`/`sharedPersistentStore = nil` — without this, downstream gates (`purgeOldHistory`, `deploySchemaIfModelChanged`) remain blocked after an account change, producing a permanently degraded state within the same app session.

## CKRecord Types & Schema
- Run `initializeCloudKitSchema()` only in DEBUG builds to deploy schema to CloudKit container.
- Store `encodeSystemFields` data with each record to preserve zone metadata and prevent infinite conflict loops.
- **2026-04-10**: `initializeCloudKitSchema(options:)` is a one-off schema-deployment tool, not a per-launch call. Gate behind a `UserDefaults` sentinel keyed to `managedObjectModel.versionIdentifiers` (with `entityVersionHashesByName` fallback, sorted+joined for determinism across `Set<AnyHashable>`) so it only runs when the model actually changes. Apple DTS guidance ("just once"), fatbobman, and multiple SO threads (68544092, 58218874) all confirm per-launch calls are expensive network round-trips that create noise in the CloudKit Development environment. Apple Forums thread 120453.

## CKShare & Participant Management
- Use zone-level sharing (CKShare per zone), not hierarchical record sharing — simpler for 2-user household.
- UICloudSharingController is UIKit — needs UIViewControllerRepresentable wrapper for SwiftUI.
- **CAUTION (iOS 17 only):** `container.share()` was reported to deadlock on iOS 17 unless called from `CKShareTransferRepresentation.prepareShare`. Tested and working on iOS 26+ with `UICloudSharingController(share:container:)` pattern. Risk accepted for iOS 26+ minimum target (Story 4-1 decision D1, 2026-04-04).
- Always use `CKContainer(identifier: "iCloud.com.wagneraz.CashOut")` — never `CKContainer.default()` — to ensure container matches `PersistenceController` config.
- Before calling `container.share()`, check `FileManager.default.ubiquityIdentityToken != nil` — if iCloud is signed out, throw a user-friendly error instead of letting `CKError.notAuthenticated` propagate.
- Reuse existing `CKShare` when re-presenting share sheet — calling `container.share(objects, to: nil)` multiple times creates duplicate CKShares in separate zones.
- Filter `CKShare.participants` for `.accepted` acceptance status when displaying partner info — pending/removed participants should not show as connected.
- App must implement `userDidAcceptCloudKitShareWith` and call `container.acceptShareInvitations(from:into:)` for partner join flow. There is NO `container.accept(metadata)` convenience method on NSPersistentCloudKitContainer.
- `.onCKShareAccepted` is NOT a real SwiftUI scene modifier — use `UIApplicationDelegate.userDidAcceptCloudKitShareWith` for share acceptance.
- **2026-04-04**: `checkSharingStatus()` must check BOTH `privatePersistentStore` and `sharedPersistentStore` — the owner's share metadata lives in the private store, but the partner's share metadata lives in the shared store. Checking only private store silently fails for the partner.
- **2026-04-04**: `extractPartnerInfo` must filter by `currentUserParticipant?.userIdentity.userRecordID` (not `role != .owner`) to find "the other person" — filtering by role breaks for the participant (they're `.privateUser`, not `.owner`, so they get themselves back instead of the owner). Guard for nil `currentUserParticipant`/`userRecordID` — when nil (identity not yet resolved), the filter passes ALL participants and the current user could be reported as their own partner.
- **2026-04-08**: `CKShare.owner` is non-optional `CKShare.Participant` on iOS 26+ — do not use optional chaining (`share.owner?`). Participant path should read partner info directly from `share.owner` (the owner IS the partner) instead of filtering `share.participants`. Owner path should use `share.owner.userIdentity.userRecordID` to identify self when filtering for the other participant. `currentUserParticipant` may be nil on locally-cached CKShare objects.
- **2026-04-09**: To cancel/delete a CKShare (owner-side), use `CKModifyRecordsOperation(recordsToSave: nil, recordIDsToDelete: [share.recordID])` on `privateCloudDatabase`. This severs the sharing relationship but does NOT delete root records. Do NOT use `purgeObjectsAndRecordsInZone` for cancel — it nukes the entire zone + managed objects.
- **2026-04-09**: `NSPersistentCloudKitContainer.share(objects, to: nil)` persists the CKShare to CloudKit BEFORE the share sheet appears. If the user dismisses the sheet without sending, an orphaned share exists. The app must either reuse it (correct pattern via cached share validation) or provide a cancel action to delete it.
- **2026-04-10**: Calling `container.share(objects, to: existingShare)` immediately after `context.save()` races the async CloudKit export the save triggers. Symptoms: "Result accumulator timeout: 3.000000, exceeded", "CoreData+CloudKit: Can't find metadata for recordID ... in zone `__defaultOwner__`" and `Metadata Inconsistency` warnings, followed by a ~3.4s total share latency. Root cause: `context.save()` stages the new record for CloudKit upload in the private default zone and registers export metadata asynchronously. `container.share()` tries to move that record into the shared zone BEFORE the export pipeline has registered its bookkeeping entry, so the internal metadata tracker points at a recordID in the wrong zone. Fix: insert `do { try await Task.sleep(nanoseconds: 800_000_000) } catch { return }` (800ms) at the top of the post-save share task (`shareNewCategoryToHousehold`, `shareNewExpenseToHousehold`) BEFORE re-fetching and calling `shareObjectsToHouseholdIfNeeded`. Must use `do/catch` not `try?` — `try?` swallows `CancellationError` and lets a cancelled task continue past the sleep. 800ms is empirically sufficient on a warm connection. Also gate on `isShareOwner == true` BEFORE the sleep — participant saves no-op inside `shareObjectsToHouseholdIfNeeded` via role guard, so the 800ms delay is wasted for participants. A more robust fix is to gate on `NSPersistentCloudKitContainer.eventChangedNotification` with `.export` completion, but there is no public API to filter events by objectID, so the short delay is the pragmatic choice.
- **2026-04-09**: `UICloudSharingController` auto-deletes the CKShare from CloudKit when the user taps "Stop Sharing" — the app only needs to clear local cached state in `cloudSharingControllerDidStopSharing`.
- **2026-04-09**: Re-presenting `UICloudSharingController(share:container:)` with a pending (unaccepted) CKShare is valid — it shows the share URL for resending. Do NOT use `init(preparationHandler:)` for existing shares — it creates duplicate CKShares.
- **2026-04-09**: `UICloudSharingController.availablePermissions` MUST include `.allowPrivate` for the "Share With More People" invitation button to appear. Without it, the controller shows only the management view (Owner + Stop Sharing) with no way to invite. For a private 2-person app: `[.allowPrivate, .allowReadWrite]`. Omitting `.allowPublic` correctly hides the "anyone with link" option.
- **2026-04-09**: In `createShare()`, the cached-share validation store must always be `privatePersistentStore` — owner-created shares live in the private database. Using `isShareOwner` to pick the store fails in solo mode (`isShareOwner == false` → picks shared store → misses the share). Also: check `isRecentlyCanceled` sentinel BEFORE `fetchShares`, not inside the success branch — a transient `fetchShares` error must not bypass the sentinel.
- **2026-04-04**: Owner and participant use different save paths for shared zone routing: Owner calls `container.share(objects, to: existingShare)` AFTER `context.save()` (post-save). Participant calls `context.assign(object, to: sharedStore)` BEFORE `context.save()` (pre-save). Both result in the expense in the shared zone. Edits/deletes need no sharing calls — `NSPersistentCloudKitContainer` handles them automatically.
- **2026-04-04**: `persistUpdatedShare` must route to the correct store based on `isShareOwner` — owner's share is in `privatePersistentStore`, partner's share is in `sharedPersistentStore`. Hardcoding `privatePersistentStore` silently fails on the partner's device.
- **2026-04-10**: Model sharing state as a `SharingState` enum (`.solo / .draft / .pending / .connected(partnerName: String?)`), not a `Bool isShared` plus sibling `partnerName: String?`. The boolean combination `isShared && partnerName == nil` cannot distinguish "draft share exists locally, user cancelled before sending" from "invite sent, awaiting acceptance" — both read as pending. The `.draft` case is load-bearing: it represents the window between `container.share(objects, to: nil)` success and either invite dispatch or cancellation. Orphan cleanup only happens when `.draft` + dismiss — never from `.pending`/`.connected` (those mean Stop Sharing, UIKit handles the remote delete). Associated value on `.connected` makes `partnerName` type-safely inaccessible in other states.
- **2026-04-10**: `UICloudSharingController.cloudSharingControllerDidSaveShare` fires on BOTH invite dispatch AND permission changes inside the sheet — it is NOT a reliable "invite sent" signal. The only ground truth is `share.participants.count > 1` (more than just the owner) on a share fetched directly from CloudKit, not from the local Core Data mirror. Per-delegate classification is fundamentally wrong; classify from participant count on a fresh fetch.
- **2026-04-10**: When fetching a just-created CKShare via `CKFetchRecordsOperation` to classify it, distinguish `CKError.unknownItem` (server authoritatively confirms the share is absent — safe to transition to `.solo`) from transient errors (`.networkUnavailable`, `.serviceUnavailable`, propagation lag — fall back to the delegate-provided share). Calling `cancelShare()` on a transient fetch failure will delete a legitimately-created share during CloudKit's 5–15s cache propagation window. Never call cancelShare from a catch-all `catch`.
- **2026-04-10**: `CKFetchRecordsOperation` can surface "record does not exist" via TWO different error shapes: (1) a direct `CKError.unknownItem` through `perRecordResultBlock`, or (2) a `CKError.partialFailure` envelope through `fetchRecordsResultBlock.failure` whose `partialErrorsByItemID` values are all `.unknownItem`. A catch clause that only checks the direct form silently falls through to the transient-error path for the wrapped form. Extract a helper (`isUnknownItemError(_:)`) that checks both cases before acting on the result. Missing this check re-introduces the orphan-CKShare bug the fetch was designed to prevent.
- **2026-04-10**: Every `.solo` state transition must be routed through a single `transitionToSolo(clearSentinel:)` helper that resets `state`, `isShareOwner`, `currentShare`, and optionally `canceledShareSentinel` in lockstep. Multiple ad-hoc sites (`finalizeShareOutcome` catch clauses, `apply` switch cases) that each set only a subset leave `isShareOwner = true` in `.solo` state — a latent hazard because `checkSharingStatus` on next launch will skip the participant branch and classify a non-existent share. The centralized helper makes it structurally impossible to forget a field.
- **2026-04-10**: `SharingState.draft` is session-local — it represents the transient window between `createShare()` success and `finalizeShareOutcome()` within a single app session, NOT a durable state worth preserving across launches. But `classify(share:isOwner:)` is a pure function that re-derives `.draft` from CKShare participant count, making the state storage-observable: any orphan share with zero invited participants (left by a crashed session or pre-fix buggy code) re-materializes as `.draft` on next launch. `checkSharingStatus` must enforce the temporal invariant by checking `currentShare == nil` BEFORE setting it — if the classify returns `.draft` AND `currentShare` was nil on entry, no in-session code created the share, so it's an orphan and must be cleaned up via `cancelShare()` (server-authoritative delete + sentinel). The mid-sheet `.draft` guard (preserve when no share found) handles the symmetric "protect in-progress draft from remote-refresh race" case but does nothing for stored orphans — that case needs its own explicit handling.
- **2026-04-10**: `scheduleBackfillIfNeeded` must be gated on `hasCommittedShare` (`.pending` or `.connected`), not called unconditionally when any owner-side share is detected. Running backfill for a `.draft` moves pre-invite expenses into the shared zone of an uncommitted share — if the user then cancels the draft (or it's an orphan being cleaned up), the expenses are left in an unshared zone with no audience. Before the SharingState refactor, `isShared=true` only ever fired for committed shares, so the guard was implicit. After the refactor, `.draft` is also an owner-side detected state and the guard must become explicit.
- **2026-04-10**: Do NOT call `clearCanceledShareSentinel()` in `checkSharingStatus`'s solo-mode fallthrough (the "no shares found" path). That path is NOT a server-authoritative confirmation — it just means `fetchActiveShare` returned nil at this instant, possibly BECAUSE the sentinel itself was suppressing a stale mirror entry. Clearing the sentinel there races the `NSPersistentCloudKitContainer` mirror reconciliation window: after `cancelShare`'s `CKModifyRecordsOperation` deletes the share server-side, the local Core Data mirror can return the stale entry for seconds to minutes until its internal recovery path processes the deletion. If we clear the sentinel prematurely, the NEXT remote-change refresh re-detects the stale mirror entry, triggers redundant orphan cleanup, and emits a `CoreData+CloudKit: Export failed ... Unknown Item ... recordChangeTag specified, but record not found` error. The sentinel is pure suppression state with zero cost to leave in place; let the TTL handle expiration. Only `createShare` on new-share success and `transitionToSolo(clearSentinel: true)` on `.unknownItem`-confirmed paths should clear it explicitly.
- **2026-04-10**: `canceledShareTTL` of 120s is tight for slow/flaky connections (e.g., travel wifi). The `NSPersistentCloudKitContainer` mirror reconciliation window after an out-of-band `CKModifyRecordsOperation` delete is undocumented and can exceed 120s. 300s is belt-and-suspenders — costs nothing because the sentinel is pure suppression state and the `wasCurrentShareNil` guard blocks false orphan detection if a real share slips past with a different recordName.
- **2026-04-10**: The `CoreData+CloudKit: Export failed ... Unknown Item ... recordChangeTag specified, but record not found` error is BENIGN when it follows an out-of-band `cancelShare` via `CKModifyRecordsOperation`. It is `NSPersistentCloudKitContainer`'s exporter trying to sync its local mirror copy of the now-deleted CKShare to CloudKit, hitting the server's "this record doesn't exist" response, and triggering its internal recovery path — which tombstones the local mirror entry. The log noise is alarming but the mirror reconciliation is proceeding correctly. There is NO documented surgical API to tell the container about an out-of-band delete: `purgeObjectsAndRecordsInZone` is zone-level nuclear, `persistUpdatedShare` requires a live CKShare, and the mirror's internal CKShare representation is not a user-accessible `NSManagedObject`. The sentinel-based filtering at `fetchActiveShare` read time is the correct mitigation.
- **2026-04-10**: `checkSharingStatus`'s startup-orphan branch must guard against re-entry during the `await cancelShare()` suspension with a `isCleaningUpOrphan: Bool` flag. Without it, a concurrent `checkSharingStatus` call (from a remote-change debounce firing while cancelShare is in-flight) sees `currentShare` already set by the outer call's preparatory assignment, so the outer call's `wasCurrentShareNil` capture is false for the inner call, the orphan guard skips, and the fall-through sets `state = .draft` briefly. The outer call's `transitionToSolo` eventually wins so the final state is correct, but the flicker breaks any future code path that observes `state` between the outer call's preparation and cancelShare completion. Pattern: `guard !isCleaningUpOrphan else { return }; isCleaningUpOrphan = true; defer { isCleaningUpOrphan = false }` at the top of the orphan branch.
- **2026-04-10**: `CKFetchRecordsOperation` wrapper must use `perRecordResultBlock` to capture results into locals and `fetchRecordsResultBlock` as the SINGLE continuation resumption point. Resuming from both blocks causes undefined behavior (double-resume); resuming from only `perRecordResultBlock` risks hanging forever if the per-record cast fails. Single-resume pattern: `fetchRecordsResultBlock` reads the locals and resumes with either the fetched share, the per-record error, or `.unknownItem`.
- **2026-04-10**: `fetchFreshShare` (and any CKShare fetch outside `persistUpdatedShare`) must route to the correct database by role: `isShareOwner ? container.privateCloudDatabase : container.sharedCloudDatabase`. Hardcoding `privateCloudDatabase` works for owner-only paths (`cancelShare`) but silently returns `.unknownItem` on the participant device for generic fetches.
- **2026-04-10**: In `checkSharingStatus`, when no share is found in either store AND current state is `.draft`, PRESERVE `.draft` — do not reset to `.solo`. A remote change notification firing during an in-progress share sheet (before the new CKShare has replicated back from CloudKit) would otherwise nuke the draft state and force a duplicate share on sheet re-present. The guard is "only transition out of .draft via explicit `createShare`/`finalizeShareOutcome` calls" — remote refreshes are advisory, not authoritative, for the draft window.

- **2026-04-07**: Seeded reference data (default categories) must use deterministic stable UUIDs — not `UUID()` — per enum case. Two CloudKit-synced devices seeding independently with random UUIDs create non-mergeable duplicates that persist permanently. Define `var stableID: UUID` with hardcoded UUID literals. Rely on client-side dedup in the repository (by name + id), NOT database-level uniqueness constraints.
- **2026-04-07**: Client-side dedup in repository fetch methods must scope to `isDefault` records only — deduplicating custom categories by name silently hides user-created records. Add secondary sort on `id` (deterministic tiebreaker within same `sortOrder`) to ensure consistent dedup winner across devices.
- **2026-04-08**: `NSPersistentCloudKitContainer` does NOT support `<uniquenessConstraints>` in the Core Data model. `loadPersistentStores` rejects the model at validation time — both private and shared stores fail with "A Core Data error occurred" (fails in <1ms, before any I/O). Apple Engineer: "This was never supported, and fails in byzantine ways, so it produces a clear error up front now" (iOS 14+, Forums thread/656380). Handle dedup in application code, not the model.
- **2026-04-08**: `context.delete()` on `viewContext` without `request.affectedStores` scoping can delete objects from the shared store — `NSPersistentCloudKitContainer` propagates the tombstone to the partner's device via CloudKit sync. Always scope delete-oriented fetches to `[privatePersistentStore]` when cleaning up private data (e.g., duplicate default categories).
- **2026-04-08**: `container.share(objects, to: nil)` shares ALL passed `NSManagedObject` instances to a new CloudKit zone. Display-layer dedup (in `fetchCategories`) does not protect this path — if 30 raw duplicate records exist in Core Data, all 30 are shared. Deduplicate managed objects before passing to `container.share()`.

- **2026-04-08**: `isShareOwner` must be a stored `var` set explicitly during `checkSharingStatus()` — a computed property reading `currentUserParticipant?.role` is unreliable because `currentUserParticipant` can be nil on locally-cached `CKShare` objects. Set `true` in the private-store path, `false` in the shared-store path, and `false` in the no-shares fallthrough.
- **2026-04-09**: Repository post-save sharing is fire-and-forget, but `container.share()` holds the main actor for its synchronous prep (~2s). Re-fetching by UUID inside `shareNewExpenseToHousehold` is acceptable — Core Data's identity map returns the in-memory object without disk I/O. **Superseded by 2026-04-10 — do NOT store the Task in the ViewModel.**
- **2026-04-10**: Post-save share tasks must be owned by `ExpenseRepository` (singleton), NOT by the ViewModel. Two reasons: (1) a `Task {}` created inside a `@MainActor` ViewModel inherits `@MainActor`, and when the outer `saveTask` in the View is doing `async let save = viewModel.saveExpense(); await save`, the Swift actor executor schedules the newly-enqueued share task ahead of the outer task's resumption — so navigation is gated on `container.share()` completing (~2s), producing a "stuck then abrupt dismiss" UX. (2) `onDisappear` on the Entry screen fires on EVERY successful save (save → `onSaveComplete` → parent dismiss → onDisappear), so a view-scoped `cancelPendingShare()` race-cancels the in-flight CloudKit share and silently drops the expense from the partner's shared zone. Fix: `ExpenseRepository.enqueueShareForNewExpense(id:)` keyed on UUID in a private dict; task body does `await Task.yield()` FIRST (releases the main actor so the outer `saveTask` resumes, navigates, dismisses), THEN `await shareNewExpenseToHousehold(id:)`, THEN prunes itself from the dict.
- **2026-04-08**: `resetState()` for session cleanup must be unconditional — no store-readiness guards. Unlike `checkSharingStatus()` which guards against mid-account-change resets, a deliberate sign-out must always clear cached sharing state (`isShared`, `isShareOwner`, `partnerName`, `currentShare`) regardless of `privatePersistentStore` availability.
- **2026-04-08**: `share.owner.userIdentity.userRecordID` is `Optional<CKRecord.ID>` even though `share.owner` is non-optional. When using it to filter participants in the owner path, always guard for nil — otherwise `nil != nil` evaluates to `false` and the owner would be included as their own partner.
- **2026-04-08**: After `acceptShareInvitations` completes in AppDelegate, call `checkSharingStatus()` to refresh partner state. Note: the completion fires before CloudKit syncs the accepted share to the local store, so this is best-effort. The `NSPersistentStoreRemoteChange` listener in ContentView provides the reliable follow-up.
- **2026-04-08**: A function declared `async throws` that has an internal `do/catch` swallowing its own errors makes the `throws` declaration misleading — callers' `catch` blocks never fire and tests asserting on thrown errors pass as false negatives. `shareObjectsToHouseholdIfNeeded` had this: the inner `do/catch` logged and returned, so the repository-layer `catch` was dead code. Fix: remove the inner `do/catch` and let errors propagate to the caller.
- **2026-04-10**: When back-filling pre-invite expenses into a fresh CKShare zone via `container.share(objects, to: share)`, filter the fetched expenses by `object.objectID.persistentStore == privateStore` first. Re-passing already-shared-zone objects races the CloudKit export cycle and produces the "Missing metadata for recordID" warning per line 94 above. Fetch object IDs on a background context, re-hydrate on viewContext, then filter.
- **2026-04-10**: CKShare backfill retry loops must classify CKError codes to avoid retry storms: only `.networkUnavailable`, `.networkFailure`, `.serviceUnavailable`, `.requestRateLimited`, `.zoneBusy` are transient (leave the marker unset so the next `checkSharingStatus` retries). All others — `.quotaExceeded`, `.permissionFailure`, `.serverRejectedRequest`, `.badContainer`, `.notAuthenticated` — and non-CKError exceptions must `markBackfillComplete` to stop the retry loop. Infinite retries on unclassified errors are worse than silent failure since every sync notification would re-trigger the attempt.

## Conflict Resolution (Last-Write-Wins)
- NSPersistentCloudKitContainer uses CKRecord change tags for framework-level LWW — NOT any custom `modifiedAt` field.
- Custom `modifiedAt` field is for display/sorting only — it does not participate in conflict arbitration.

## Offline Queue & Background Sync
- NSPersistentCloudKitContainer manages offline queue, retry logic, and change token persistence internally.
- Hard delete propagation works via NSPersistentHistoryTracking — but tombstone window expiry can leave orphaned records on offline partner.
- Purge NSPersistentHistoryTransaction entries older than 7 days on app launch to prevent unbounded growth.
- **2026-04-10**: `NSPersistentHistoryChangeRequest.deleteHistory(before:)` must set `affectedStores = [privatePersistentStore]` explicitly — unscoped purges touch the shared store's history, which `NSPersistentCloudKitContainer` uses for its own export-tracking machinery. Client-side purges of shared-store history can trigger phantom re-exports and duplicate tombstones.

## CKSubscription & Real-Time Updates
- Do NOT create manual CKDatabaseSubscription when using NSPersistentCloudKitContainer — the framework manages its own subscriptions internally.
- Creating a separate subscription causes double-processing, token conflicts, and missed updates.
- Use `.NSPersistentStoreRemoteChange` notification to detect partner changes, not manual CKFetchDatabaseChangesOperation.
- Must enable Background Modes → Remote Notifications capability for silent push.
- **2026-04-04**: `NSPersistentCloudKitContainer.eventChangedNotification` (sync lifecycle: success/failure tracking) is distinct from `.NSPersistentStoreRemoteChange` (data arrival). Use `eventChangedNotification` for sync health monitoring, `.NSPersistentStoreRemoteChange` for triggering data refreshes. They coexist safely — different notification names, different purposes. `eventChangedNotification` is the ONLY official API for monitoring sync health (iOS 14+).
- **2026-04-04**: Sync failure detection windows (`lastSuccessDate` + threshold) must initialize to `.distantPast`, not `Date()` — `Date()` creates a hidden 5-minute grace period on cold launch where failures won't trigger `.syncFailure` even if the device was offline for hours. Guard failure-counting against `.noICloudAccount` state — CloudKit event failures should not overwrite the root cause (no account) with a less informative `.syncFailure`.

## Security & Zone Permissions
- Observe CKAccountChanged notification to detect iCloud account change — flush cached tokens and reconcile local state.
- Household data must never be stored in publicCloudDatabase — only private + shared scopes.

## Known Bugs
- **iOS 18+ data-loss bug (ACTIVE, no fix):** When iCloud is disabled/signed out, NSPersistentCloudKitContainer can delete local data on first init (Apple Forums thread 772015). Guard with `FileManager.default.ubiquityIdentityToken != nil` check before setting cloudKitContainerOptions.
- **mergePolicy:** Use `NSMergeByPropertyStoreTrumpMergePolicy` (not ObjectTrump) on viewContext — lets the persistent store's CloudKit-merged state win over stale in-memory copies.
- All Core Data attributes MUST be Optional in the model editor — NSPersistentCloudKitContainer cannot sync non-optional attributes (silent failure).
- UUID attributes must have "Uses Scalar Type" UNCHECKED in Xcode model editor.
- `CKSharingSupported` Info.plist key must be added manually — not in Xcode autocomplete dropdown.
- Deploy schema from Development to Production in CloudKit Console before TestFlight — `initializeCloudKitSchema()` only pushes to Development.
- `usesScalarValueType` in .xcdatamodel does NOT affect CloudKit sync — it only controls Swift codegen (Bool vs NSNumber?). What CloudKit requires is `optional="YES"` on the attribute in the model editor. Scalar types with `optional="YES"` are valid for CloudKit sync.
- The iOS 18+ iCloud data-loss guard (`ubiquityIdentityToken == nil`) must protect BOTH the private AND shared store descriptions — don't nil out only the private store's cloudKitContainerOptions while leaving the shared store's options active.
- Must call `UIApplication.shared.registerForRemoteNotifications()` in `didFinishLaunchingWithOptions` — without it, silent push for CloudKit sync is never delivered.
- `NSPersistentCloudKitContainer` has NO `handleRemoteNotification` method. It auto-processes silent pushes via `NSPersistentStoreRemoteChangeNotificationPostOptionKey`. The `didReceiveRemoteNotification` delegate just needs to call `completionHandler(.newData)`.
- Explicitly set `cloudKitContainerOptions` with `containerIdentifier` on BOTH private and shared store descriptions — do not rely on auto-detection from the model/entitlements.
- **2026-04-09**: `NSPredicate(format: "id IN %@", uuids.map { $0 as CVarArg })` silently matches nothing — `UUID as CVarArg` produces `__SwiftValue`, not `NSUUID`. Use `uuids as [NSUUID]` for the correct ObjC bridge type in Core Data predicates.
- **2026-04-09**: To propagate a category delete to the partner, delete from the private store only — `NSPersistentCloudKitContainer` generates a tombstone via history tracking. The shared store is a framework-managed mirror; deleting from it discards a local copy that the framework will re-import on next sync.
- **2026-04-09**: `reorderCategories` must scope fetches to `[privatePersistentStore]` — unscoped fetches touch records in both stores, generating unnecessary CloudKit traffic and causing LWW collisions if both users reorder independently. Per-user display order lives in UserDefaults (not synced); Core Data `sortOrder` is the canonical fallback.
- **2026-04-11**: READ fetches on `viewContext` are ALSO affected by the multi-store aggregation rule — not just write/delete operations. An unscoped `NSFetchRequest` on `NSPersistentCloudKitContainer` with both `privatePersistentStore` and `sharedPersistentStore` loaded returns records from BOTH stores, because the container mirrors private-store data into the shared-store backing zone. For `fetchCategories`, this surfaced as `found 12 (pre-dedup) → returning 6 (post-dedup)` on every call (6 defaults × 2 copies), with a dedup block masking the duplication as a "safety net." Pattern to fix: role-based `affectedStores` scoping (solo + owner → `[privateStore]`; participant + `.connected` → `[sharedStore]`) mirroring `ExpenseRepository.deleteExpense`'s store selection. The dedup block should be KEPT as genuine defense-in-depth against participant-mode shared-zone import races, but scoping eliminates the 2N duplication at the source and halves fetch work. FRC fetches for expenses do NOT suffer the same issue because each expense is authoritative in exactly one store per device (owner writes go to private via `container.share`; participant writes go to shared via `prepareObjectForSharedSave`); the double-insert path was specific to the default-category seeding flow.
- **2026-04-11**: Share-enqueue guards (`state != .solo`) belong at the REPOSITORY call site (`enqueueShareForNew…`), NOT inside the worker function (`shareNewExpenseToHousehold`). The worker contains an 800ms `Task.sleep` for the CloudKit metadata-race fix, so a no-op worker still allocates a Task, writes to `activeShareTasks[id]`, and burns 800ms of scheduler time per call. In solo mode this was ~15+ wasted Tasks per interactive session. Keep the inner `isShareOwner` guard as defense-in-depth for participant role (catches a different case than solo state), but short-circuit as early as possible. Same pattern for `ContentView`'s `NSPersistentStoreRemoteChange` listener: add `guard state != .solo` INSIDE the debounce Task AFTER the 500ms sleep (early-return before calling `checkSharingStatus`). Every path that transitions OUT of `.solo` bypasses that listener entirely — the guard is safe.
- **2026-04-08**: `seedDefaultCategoriesIfNeeded` count query MUST use `affectedStores = [privateStore]` — without it, shared-store categories from partner sync satisfy the guard and seeding into the private store is silently skipped. After DEBUG destructive migration, CloudKit reimports records with new objectIDs creating duplicates (42 = 7×6 observed). Fix: scope count to private store + predicate `isDefault == YES`.
- **2026-04-08**: `container.share(objects, to: share)` after `context.save()` causes "Missing metadata for recordID" — the save triggers async CloudKit export which races with the share call. The share still succeeds, but the metadata warning indicates internal bookkeeping confusion. For owner path, consider gating share on `NSPersistentCloudKitContainer.eventChangedNotification` export completion instead of immediate fire-and-forget.
- **2026-04-09**: After deleting a CKShare via `CKModifyRecordsOperation`, the local Core Data mirror (`fetchShares(in:)`) still returns the stale share until NSPersistentCloudKitContainer processes the deletion (5–120s, or never if offline). `checkSharingStatus()` must use a sentinel (`canceledShareRecordName` + timestamp in UserDefaults) to skip stale shares within a TTL window. The sentinel must also gate `createShare()`'s validation path to prevent reusing a deleted share. Clear the sentinel when: (a) no shares found in stores (mirror caught up), or (b) a new share is created.
